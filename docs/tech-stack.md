# Tech Stack Document — Mollie CLI

## Language

**Go 1.22+**

Go is the natural choice for a developer CLI:
- Compiles to a single, self-contained binary with no runtime dependencies — easy to distribute and install.
- Excellent standard library (`net/http`, `os`, `encoding/json`, `text/tabwriter`) covers most CLI needs without pulling in large dependencies.
- Strong cross-compilation support lets us ship binaries for macOS (arm64/amd64), Linux (amd64/arm64), and Windows from one build pipeline.
- The team behind the official Mollie Go SDK is already producing Go code, creating consistency.

Minimum version is `1.22` to take advantage of the improved `net/http` routing and range-over-integer syntax. The `go.mod` file will enforce this.

---

## API Client

**`github.com/mollie/mollie-api-golang`** (official Mollie Go SDK, generated by Speakeasy)

This SDK is the official, Mollie-maintained Go client. Key properties relevant to the CLI:

| Property | Detail |
|---|---|
| Auth | HTTP Bearer token via `components.Security{APIKey: ...}` |
| Test mode | Global `client.WithTestmode(bool)` option; defaults to `true` — pass `--live` to disable |
| Pagination | Cursor-based; responses expose a `.Next()` method |
| Retries | Built-in configurable exponential backoff |
| Error types | Typed errors (`apierrors.ErrorResponse`, `apierrors.APIError`) |
| Profile ID | Global `client.WithProfileID(string)` option |

The CLI initialises a single SDK client instance per invocation, injecting the API key, test-mode flag, and optional profile ID from the resolved configuration.

## CLI Framework

**[`github.com/spf13/cobra`](https://github.com/spf13/cobra)**

Cobra is the de facto standard for Go CLIs (used by kubectl, GitHub CLI, Hugo, and many others). It provides:
- Hierarchical subcommand structure (`mollie payments create`, `mollie customers list`, etc.).
- Persistent flags that are inherited by all subcommands (used for `--output`, `--test`, `--profile`, `--api-key`).
- Built-in `--help` generation from command and flag descriptions.
- Native `completion` command generation for bash, zsh, fish, and PowerShell (used in Milestone 7).
- Well-understood conventions so contributors can onboard quickly.

## Configuration Management

**[`github.com/spf13/viper`](https://github.com/spf13/viper)**

Viper integrates natively with Cobra and provides:
- TOML/YAML/JSON config file reading.
- Automatic environment variable binding (e.g. `MOLLIE_API_KEY`, `MOLLIE_LIVE_MODE`).
- A clear precedence chain: explicit flag → environment variable → config file → default.

**Config file location**

The config file path follows the [XDG Base Directory Specification](https://specifications.freedesktop.org/basedir-spec/basedir-spec-latest.html):

```
$XDG_CONFIG_HOME/mollie-cli/config.toml
# fallback:
~/.config/mollie-cli/config.toml
```

The file is created with mode `0600` (owner read/write only) to protect the API key at rest.

**Example config file shape**

```toml
api_key     = "access_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
profile_id  = ""            # optional; required for org-level tokens
live_mode   = false         # default is test mode; set to true or pass --live for live environment
output      = "table"       # table | json | yaml
```

## Interactive Prompts

**[`github.com/charmbracelet/huh`](https://github.com/charmbracelet/huh)**

`huh` is a modern, composable TUI form library from Charm. It is used for:
- The first-run `auth setup` wizard.
- Interactive create/update flows when required flags are not supplied on the command line.
- Confirmation prompts before destructive actions.

`huh` was chosen over the older `survey` library because it is actively maintained under the Charm umbrella (same ecosystem as `lipgloss` and `bubbletea`), has a clean API, and supports accessible plain-text fallback for non-TTY environments.

## Output & Formatting

### Tables — `text/tabwriter` (stdlib) + **[`github.com/charmbracelet/lipgloss`](https://github.com/charmbracelet/lipgloss)**

For human-readable table output:
- `text/tabwriter` handles column alignment efficiently.
- `lipgloss` is used for ANSI colour styling (header rows, status badges, highlighted IDs).
- Colour output is automatically disabled when stdout is not a TTY or when `--no-color` is set.

### JSON — `encoding/json` (stdlib)

JSON output uses `json.MarshalIndent` for pretty-printed, pipeable output.

### YAML — **[`gopkg.in/yaml.v3`](https://github.com/go-yaml/yaml)**

YAML output (Milestone 7) uses the standard `yaml.v3` library for marshalling.

## Error Handling & Logging

- Errors from the Mollie SDK are inspected with `errors.As` to distinguish typed API errors (`apierrors.ErrorResponse`) from transport-level errors.
- User-facing errors are printed to `stderr` with a consistent format: `Error: <message>`.
- The exit code is `1` on any error, `0` on success (following Unix conventions).
- Verbose debug logging (`--verbose` / `MOLLIE_DEBUG=true`) uses Go's standard `log/slog` package (available since Go 1.21) writing structured JSON lines to stderr.

## Testing

| Layer | Tool |
|---|---|
| Unit tests | `testing` (stdlib) |
| Assertions | [`github.com/stretchr/testify`](https://github.com/stretchr/testify) (`assert`, `require`) |
| HTTP mocking | [`net/http/httptest`](https://pkg.go.dev/net/http/httptest) (stdlib) |
| CLI integration tests | Cobra's `ExecuteC` with captured stdout/stderr buffers |

The Mollie SDK client accepts a custom `http.Client`, which makes it straightforward to inject a test server without any additional mocking framework.

## Build & Distribution

**[`github.com/goreleaser/goreleaser`](https://github.com/goreleaser/goreleaser)**

GoReleaser automates the full release pipeline:
- Cross-compilation for `darwin/amd64`, `darwin/arm64`, `linux/amd64`, `linux/arm64`, `windows/amd64`.
- Generates compressed archives (`.tar.gz`, `.zip`) and checksums.
- Publishes GitHub Releases with auto-generated changelogs.
- Optionally publishes a [Homebrew tap](https://brew.sh/) formula for macOS installation via `brew install mollie-cli`.

**CI** runs on GitHub Actions:
- `go vet`, `staticcheck`, and `golangci-lint` on every pull request.
- Full test suite on every push to `main`.
- GoReleaser triggered on version tags (`v*`).

├── internal/

The `cmd/` package contains only command wiring and flag parsing. All business logic (API calls, formatting, config I/O) lives in `internal/` so it can be tested in isolation without invoking Cobra.
